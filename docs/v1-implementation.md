# V1 实施文档

本文档定义第一版的实施边界，目标是替代神医助手在本地媒体库中的核心生成职责。

## 1. 目标

V1 的核心目标不是做一个大而全的媒体平台，而是稳定接管以下工作：

- 监听媒体目录中新落盘的视频文件
- 判断文件何时可安全处理
- 生成字幕、媒体信息、NFO、BIF 等神医助手当前负责的关键伴生文件
- 提供 Web 端配置、日志、重试、手动补扫能力

V1 不接管以下职责：

- 下载任务管理
- BT/订阅管理
- Emby 媒体库管理
- 多站点复杂刮削

## 2. 现有链路与替代范围

当前链路可抽象为：

1. `AniRss` 基于 `TMDB` 等数据整理目录与文件名
2. `qBittorrent` 下载文件
3. 文件被复制或落盘到 Emby 可见目录
4. 神医助手生成 NFO、媒体信息、BIF 及部分增强元数据，图片是否由它生成需视 Emby 配置共同判断
5. `Emby` 扫描并消费这些产物

V1 替代步骤：

1. 保留 `AniRss` 与 `qBittorrent`
2. 保留 `Emby`
3. 用本项目替代神医助手的“落盘后生成”职责
4. 在并行验证稳定后，停用神医助手对应功能

## 3. V1 功能范围

### 3.1 必做

- 多目录监听
- 启动全量补扫
- 文件稳定性检测
- 任务去重与重试
- 字幕提取
- `mediainfo.json` 生成
- 基于 `TMDB` 的 `nfo` 生成
- actor/外部 ID/剧集组等增强元数据写入
- BIF 生成
- Web 管理端
- 手动重扫/重试
- 运行日志与错误记录

### 3.2 可选但不阻塞 V1

- 单集 `thumb.jpg` 本地抽帧兜底
- OCR 字幕转换
- Emby API 集成
- 插件化工具链

### 3.3 明确不做

- 替代 `AniRss` 的订阅、命名、数据库映射逻辑
- 支持大量刮削源站点
- 完整复刻神医助手所有历史行为
- 远端网盘直链场景
- 默认接管 Emby 已能稳定保存到媒体目录的图片下载职责

## 4. 产物定义

### 4.1 字幕

输入容器：

- `mkv`
- `mp4`
- `ts`
- 其他经底层工具验证可提取字幕轨的容器

输出命名：

`视频文件名.语言.备注或字幕名称.字幕格式`

规则：

- 优先保留原始轨道格式
- 文本字幕可直接导出为 `srt`、`ass` 等
- 图片字幕保留为 `sup`、`sub/idx` 等原始格式
- 若无法安全转换，则不做伪转换
- 若可转但损失明显，则默认保留原格式，转换能力后续再开关化

### 4.2 媒体信息

输出文件：

`视频文件名-mediainfo.json`

来源优先级：

1. `mediainfo`
2. `ffprobe`

策略：

- 尽量保留原始字段
- 再补一层统一字段，便于 Web 端展示和后续 NFO 生成

### 4.3 抽帧图

这一类图片不依赖刮削，直接从本地视频里抽帧生成。

用途定位：

- 作为本地预览图
- 作为单集图片缺少远程剧照时的兜底图
- 与 `TMDB` 的 poster/backdrop/still 不是同一类产物

输出文件：

`视频文件名-thumb.jpg`

默认策略：

- 取视频 50% 位置抽一帧
- 抽帧失败时回退到 30%
- 可配置输出宽度

补充说明：

- Emby 对单集图片的常见命名就是 `{name}-thumb.ext`
- 所以单集最终落地文件仍建议统一为 `视频文件名-thumb.jpg`
- 这个文件的来源可以是本地抽帧，也可以是远程刮削到的当集剧照
- 如果没有拿到远程当集剧照，就回退到本地抽帧结果

V1 决策：

- 默认不把 `thumb.jpg` 作为主交付项
- 只有当后续验证发现单集图确实需要本项目补齐时，才启用这一能力
- 因为当前 Emby 已开启图片保存和预下载，图片能力应优先复用 Emby

### 4.4 刮削图片

这一类图片依赖联网刮削，和 `thumb` 是两条不同链路。

图片来源：

- `TMDB` 作为主数据源
- 可选补充源：`fanart.tv`

V1 目标：

- 电影：poster、backdrop
- 剧集/番剧：series poster、season poster、episode still

命名策略：

- 先按 Emby 常见伴生文件规则设计
- 具体命名在实现前先拿样本目录实测后定稿

当前建议的 Emby 兼容落地规则：

- 剧集根目录：`poster.jpg`、`backdrop.jpg`
- 季度图片：`seasonXX-poster.jpg`
- 单集图片：保存为对应视频同名的 `视频文件名-thumb.jpg`

也就是说：

- 单集一般不用 `poster`
- 单集如果有 `TMDB` 当集图片，应该优先把它落成 `-thumb.jpg`
- 单集如果没有可用远程图片，再回退本地抽帧生成 `-thumb.jpg`

来源策略说明：

- 从你提供的样本看，当前链路里不仅有 `poster.jpg`、`fanart.jpg`，还有 `clearlogo.png`、`clearart.png`
- `poster/backdrop(or fanart)/episode still` 用 `TMDB` 基本有希望覆盖
- 但 `clearlogo/clearart` 更像 `fanart.tv` 这类美术资源站提供的资产类型
- 因此“只接 `TMDB`”大概率足够覆盖 NFO、海报、背景图、单集剧照
- 但如果要尽量复刻你现在的图片产物集合，后续多半还要补 `fanart.tv`
- 所以 V1 建议分层处理：
- 第一层：先做 `TMDB`，覆盖主流程
- 第二层：后续按需补 `fanart.tv`，专门提供 `clearlogo/clearart`

V1 决策：

- 图片下载链路先不作为第一版主线交付
- 先观察 Emby 对 `poster/fanart/clearlogo/clearart` 的现有能力是否已足够
- 文档保留设计位，后续如有必要再接管

当前判断：

- 你现在无法进入神医助手配置页，所以暂时无法确认它对“图片”到底是抽帧图、剧照图还是两者都生成
- 因此 V1 文档里先把这两类能力拆开设计
- 真正实现前，最好拿一个现有产物目录做样本核对，再决定默认开启哪些图片类型

### 4.5 NFO

输出文件：

`视频文件名.nfo`

数据来源：

- 文件名/目录名中的结构化信息
- `TMDB` 刮削结果
- 本地探测出的媒体信息

重点增强字段：

- actor / role
- `tmdbid` / `tvdbid` / `imdb_id`
- `uniqueid`
- 剧集组、显示顺序等剧集增强信息

V1 范围：

- 生成 Emby 可消费的基础 NFO
- 优先覆盖番剧/剧集场景
- 不追求一次支持所有特殊条目
- 但要把神医助手当前明显在做的 actor 与外部 ID 增强纳入范围

### 4.6 BIF

V1 中的处理方式：

- 保留设计和接口位置
- 作为 V1 主线交付项之一
- 先做格式与 Emby 兼容性验证
- 默认命名沿用现有样本：`视频文件名-320-10.bif`

## 5. 技术选型

### 5.1 后端

- 语言：`Go`
- HTTP：Go 原生或轻量路由框架
- 配置：`YAML`
- 持久化：`SQLite`

理由：

- 单文件部署简单
- 适合长期常驻
- 调用外部工具方便
- 任务并发和文件监听适配度高

数据库选择说明：

- `SQLite` 的确是“单写多读”模型，但这对 V1 不是问题
- 本项目是单机、单服务进程为主，不是高并发多节点系统
- 任务状态写入频率很低，远小于 Web 查询和日志查看频率
- 开启 `WAL` 后，并发读体验会比默认模式更适合这种后台工具
- 因此 V1 继续用 `SQLite`，能显著降低部署复杂度
- 如果未来要拆成多进程写入、远程节点协作，或者任务量远超预期，再迁移 `PostgreSQL`

### 5.2 前端

- `React`
- `TypeScript`
- `Vite`
- 组件库可后选，但建议先上成熟表格与表单组件

理由：

- 适合配置型后台
- 便于 AI 协作产出页面
- 后续状态管理、日志页、任务页扩展方便

### 5.3 外部依赖工具

- `ffmpeg`
- `ffprobe`
- `mkvextract`
- `mediainfo`

V1 要求：

- Web 端可配置路径
- 启动时执行依赖检查
- 页面可查看检测结果

## 6. 系统架构

逻辑模块：

1. `watcher`
2. `scanner`
3. `stability-checker`
4. `scheduler`
5. `task-runner`
6. `scraper`
7. `artifact-writer`
8. `store`
9. `api`
10. `web`

职责说明：

- `watcher`：监听目录变化
- `scanner`：启动补扫、过滤扩展名、识别候选文件
- `stability-checker`：判断文件是否仍在写入
- `scheduler`：生成任务、去重、排队、限流
- `task-runner`：调用外部工具执行本地提取
- `scraper`：访问 `TMDB` 获取元数据与增强信息
- `artifact-writer`：落地字幕、NFO、BIF、JSON 及可选图片产物
- `store`：保存配置、任务、错误、缓存
- `api`：提供配置与任务接口
- `web`：React 管理端

## 7. 处理流程

### 7.1 自动处理流程

1. 监听到新文件或文件变更
2. 过滤非视频文件
3. 进入稳定性检测
4. 文件稳定后创建媒体任务
5. 执行本地探测
6. 按配置执行字幕提取、媒体信息生成、BIF 生成
7. 执行 `TMDB` 匹配与增强信息刮削
8. 生成 NFO，并按需补充图片兜底产物
9. 记录任务结果

### 7.2 手动补扫流程

1. 用户在 Web 端选择目录或单文件
2. 生成补扫任务
3. 复用同一条处理链路

## 8. 稳定性设计

### 8.1 文件稳定检测

默认策略：

- 首次发现后延迟一段静默时间
- 连续两次检查文件大小不再变化才进入处理
- 若文件被占用或读取失败，则延迟重试

### 8.2 幂等与去重

- 同一路径同一版本文件只允许一个进行中的任务
- 文件修改时间或大小变化后可重新入队
- 已生成产物是否覆盖由规则控制

### 8.3 错误处理

- 外部工具失败要保留原始 stderr
- `TMDB` 请求失败要记录响应摘要
- 失败任务支持手动重试
- 可按任务类型配置自动重试次数

## 9. 配置模型

建议配置项：

- 监听目录列表
- 扩展名白名单
- 工具路径配置
- 任务开关
- 覆盖策略
- 并发数
- 静默时间
- 稳定性检测次数
- `TMDB API Key`
- 图片接管策略
- NFO 生成策略
- 日志级别

建议保存在：

- 主配置：`config.yaml`
- 运行数据：`SQLite`

数据库运行建议：

- 启用 `WAL` 模式
- 设置合理的 `busy_timeout`
- 所有写操作通过后端服务统一串行化或小范围事务化
- 不允许多个独立进程同时直接写同一个库文件

## 12. 数据库草案

核心表：

- `watch_dirs`
- `media_files`
- `tasks`
- `task_logs`
- `artifacts`
- `tool_status`
- `scrape_cache`

最小字段建议：

- `media_files`：路径、大小、修改时间、哈希摘要、最后处理时间
- `tasks`：任务类型、状态、重试次数、开始时间、结束时间、错误摘要
- `artifacts`：产物类型、路径、生成时间、来源任务

建议补充字段：

- `artifacts`：增加 `source` 字段，标记来源于 `generated`、`emby`、`external`
- `scrape_cache`：增加 `payload_type`，区分 `series`、`season`、`episode`、`credits`

数据库范围约束：

- V1 的数据库不是媒体主库，也不是“把所有番剧/电影条目完整镜像到本地”
- 默认只保存运行所需的信息、少量关联索引和轻量刮削缓存
- 不走“像隔壁 strm 工具那样长期积累一大堆业务 entry 元数据”的路线
- 真正的权威媒体元数据仍以外部源和最终生成的 `nfo` 为准

建议实际存储内容：

- `watch_dirs`：监控目录配置
- `media_files`：本地文件观测状态
- `tasks`：任务调度与执行状态
- `task_logs`：任务日志
- `artifacts`：已经生成的伴生文件清单
- `tool_status`：工具探测结果
- `scrape_cache`：近期刮削结果缓存

`scrape_cache` 建议只存：

- 外部源类型，如 `tmdb`、`fanarttv`
- 外部 ID，如 `tmdb_id`、`tvdb_id`
- 请求参数摘要，如语言、season、episode
- 精简后的响应 JSON 或原始响应摘要
- 缓存过期时间

不建议在 V1 存的内容：

- 全量演员、剧集、季度、集条目明细主表
- 全量图片 URL 历史版本
- 长期积累的远程站点镜像数据
- 复杂的 cross-library 媒体关系图

一句话说：

- 这个库主要存“任务和状态”
- 不是存“完整媒体世界模型”

## 14. 验收标准

达到以下条件即可视为 V1 可试运行：

- 新落盘视频能自动入队
- 文件未写完时不会误处理
- 能稳定生成 `mediainfo.json`
- 能稳定抽取字幕或明确记录不支持原因
- 能稳定生成 `bif`
- 能从 `TMDB` 拉取基础元数据并生成带 actor/ID 增强的 `nfo`
- Web 端可以完成配置、查看任务、失败重试
- 与神医助手并行运行时，产物结果可对比验证
