# NyaMediaMetadataTool

面向本地媒体库的元数据与伴生文件生成工具，目标是逐步替代神医助手在本地链路中的职责。

目标是监控指定目录，按规则为视频文件生成或抽取以下内容，并最终接管神医助手当前负责的本地生成流程：

- 字幕文件
- BIF 预览索引
- `mediainfo.json`
- `nfo`
- 可选图片增强能力

同时提供一个 Web 端，用于配置监控目录、启用的工具链、命名规则与任务状态查看。

## 1. 当前需求整理

### 1.1 监控能力

- 支持配置多个监控目录
- 支持递归扫描子目录
- 支持启动后全量补扫
- 支持文件新增、修改后的增量处理
- 支持过滤扩展名，当前重点是“只要能提取字幕/媒体信息的视频容器”，例如 `mkv`、`mp4`、`ts`

### 1.2 目标产物

#### 字幕提取

输入：不限于 `mkv`、`mp4`，凡是底层工具可识别并可导出字幕轨的容器都可纳入，例如 `ts`

输出命名规则：

`视频文件名.语言.备注或字幕名称.字幕格式`

示例：

- `Movie.zh.ass`
- `Movie.jpn.forced.srt`
- `Movie.eng.Commentary.sup`

说明：

- `语言` 建议统一为 ISO 639-2 / 639-3 或 Emby 常见识别值，避免中英混写
- `备注` 可来自字幕轨标题、forced、default、commentary 等标记
- 对于无语言、无标题的轨道，需要定义兜底规则，否则命名会不稳定
- 图片字幕如 `sup` 是否作为外挂字幕被 Emby 稳定识别，需要单独验证。第一版可以支持“导出原始格式”，但是否默认生成应做成可配置项

#### BIF 提取

输出命名建议：

`视频文件名-{宽度}-{间隔秒}.bif`

示例：

- `Movie-320-10.bif`

说明：

- 这里的 `320` 更适合表示缩略图宽度，而不是分辨率全称
- `10` 建议定义为“每 10 秒取一张图”
- BIF 兼容性需要与目标播放器/Emby 实测对齐

#### 媒体信息提取

输出命名：

`视频文件名-mediainfo.json`

建议内容来源：

- `MediaInfo` CLI
- 或 `ffprobe` 输出后做统一结构转换

#### 缩略图

输出命名：

`视频文件名-thumb.jpg`

建议规则：

- 默认从视频中间位置取帧
- 可配置偏移策略，如 10%、30%、50%
- 可增加“避开纯黑帧”的简单检测

当前判断：

- 如果 Emby 已开启“将媒体图片保存到媒体文件夹中”和“预先下载图像”，剧集海报、背景图、logo、clearart 很可能已经由 Emby 负责落盘
- 因此本项目第一版不默认接管图片下载职责
- `thumb.jpg` 仍保留设计位，作为单集图缺失时的本地兜底能力

#### NFO

输出命名：

`视频文件名.nfo`

说明：

- 需要先明确 NFO 来源
- 如果只是把现有媒体信息转成本地 NFO，这是“导出”
- 如果要联网刮削电影/剧集元数据，当前优先考虑 `TMDB`，但这会引入匹配、缓存、限流、失败重试等额外复杂度
- 从现有样本与神医助手说明看，`nfo` 中的 actor、`tmdbid`、`tvdbid`、剧集组等增强信息大概率不是 Emby 默认写出的，需要本项目接管

## 2. 推荐产品形态

### 2.1 第一阶段建议：独立服务

优先做成独立服务，而不是一开始就做 Emby 插件；目标上明确是替代神医助手，但替代方式应是“先独立服务化，再逐项接管能力”，而不是直接复制一个 Emby 插件。

推荐形态：

- 后端常驻进程
- 本地 Web 管理端
- 通过外部 CLI 工具完成实际抽取与生成

原因：

- 开发与调试成本更低
- 不被 Emby 插件接口和运行环境强绑定
- 以后也能服务 Jellyfin、Kodi、本地整理流程
- 工具链更容易替换，不会被某个插件生态卡住
- 你现在本地链路已经包含 `AniRss`、`Emby`、`qBittorrent`、神医助手，独立服务更适合先接管“落盘后的生成任务”，逐步替掉神医助手，而不是一次性推倒重来

### 2.2 第二阶段可选：Emby 集成

等独立服务稳定后，再考虑增加 Emby 集成：

- 方案 A：Emby 插件只负责触发任务，核心逻辑仍在独立服务中
- 方案 B：独立服务监听媒体目录，不依赖 Emby 触发

更推荐方案 B，插件只做增强，不做核心依赖。

补充判断：

- 你当前下载目录、文件名、数据库映射已经由 `AniRss + TMDB` 整理得比较规范，这是很有利的前提
- 说明这个工具不一定要接管“下载与命名”，更适合优先接管“落盘后的生成任务编排”
- 如果未来要补足神医助手的能力，可以参考其现有功能，但实现上按模块拆开，避免一次把所有历史包袱都背进来
- 替代目标不是复刻它的页面和实现，而是把你现在依赖它生成的结果稳定产出来
- 图片这部分是否接管，需要以 Emby 现有能力是否满足为准；`nfo / mediainfo / bif / 字幕 / TMDB增强` 则应作为优先接管范围

## 3. 开发语言选择

## 推荐：Go

当前仓库已经有 `go.mod`，继续使用 Go 最合适。

### 3.1 选择 Go 的原因

- 很适合做文件监控、任务队列、并发执行
- 生成单文件部署简单，适合家庭服务器、NAS、Windows 小主机
- 调用外部工具方便，封装 CLI 成本低
- 自带 HTTP 服务能力，做 Web 后台足够轻量
- 内存和启动速度通常优于 Java / Node 大型框架方案

### 3.2 不优先选的方案

#### C# / Emby 插件

优点：

- 更贴近 Emby 插件生态

问题：

- 会更早被绑定到 Emby
- 插件生命周期、API、部署、调试都更受限制
- 如果以后想脱离 Emby，迁移成本高

#### Python

优点：

- 原型快

问题：

- 部署和依赖管理不如 Go 干净
- 长期做常驻服务和跨平台分发时体验一般

#### Node.js

优点：

- 做前端后台一体方便

问题：

- 纯后端任务编排与单文件分发不如 Go 顺手
- 如果前端并不复杂，会显得偏重

## 4. Web 端建议

需要 Web 端，而且建议一开始就做。

第一版仍建议尽快做 Web 端，但前端改为 React 技术栈，而不是服务端模板页面。

- Go 后端提供 HTTP API
- 前端使用 React 框架
- 首选 `Vite + React + TypeScript`
- UI 层可选一套成熟组件库，先保证配置、日志、任务表格体验

这样可以兼顾后续可维护性和现有使用习惯，不必为了“先快一点”牺牲管理端体验。

### 4.1 Web 端首批页面

- 仪表盘：首页，展示监控目录、任务状态、失败记录
- 目录配置：新增、编辑、删除监控目录
- 工具配置：配置 `ffmpeg`、`ffprobe`、`mkvextract`、`mediainfo` 等路径
- 规则配置：选择生成哪些产物、命名规则、覆盖策略
- 任务记录：最近处理文件、成功/失败、错误日志
- 手动任务：手动重扫目录或重处理单个文件

### 4.2 Web 端首批配置项

- 监控目录列表
- 支持的视频扩展名
- 输出项目开关：字幕/BIF/mediainfo/thumb/NFO
- 是否覆盖已有文件
- 并发数
- 失败重试次数
- 外部工具路径
- 命名模板
- 日志级别

## 5. 技术方案草图

### 5.1 模块划分

- `watcher`：目录监控与启动补扫
- `scanner`：识别视频文件和需要处理的任务
- `queue`：任务排队、去重、重试
- `runner`：调用外部工具
- `metadata`：统一产物描述与命名
- `web`：配置与管理页面
- `store`：本地配置和任务状态持久化

### 5.2 外部工具建议

#### 字幕

- `mkvextract`：适合 MKV
- `ffmpeg`：可作为通用兜底
- MP4 字幕轨支持需要单独验证，尤其是 `mov_text`、`tx3g` 转出格式问题

#### BIF

- `ffmpeg` 抽帧
- 自己拼装 BIF 文件，或研究是否有现成 CLI

#### mediainfo

- 首选 `mediainfo --Output=JSON`
- 备选 `ffprobe -print_format json`

#### thumb

- `ffmpeg -ss ... -frames:v 1`

补充说明：

- 这里的 `thumb` 更偏向“本地从视频抽帧生成的缩略图”
- 如果你希望的是剧集/番剧的封面图，那通常不是抽帧，而是从 `TMDB` 之类的元数据源获取 poster/backdrop/still
- 因此这里实际上应区分两类图片：`视频抽帧图` 与 `刮削元数据图`

#### NFO

- 如果是本地导出，可自己生成 XML
- 如果涉及联网刮削，当前优先考虑 `TMDB`，并需要定义站点映射、匹配规则、缓存策略
- 结合神医助手现有说明，后续还要考虑 actor 增强、剧集组、备选语言、原语言海报等增强逻辑，但不要求第一版全部做齐

## 6. 已知问题与风险

### 6.1 字幕提取并不统一

这是最容易踩坑的部分之一：

- MKV 和 MP4 的字幕轨格式差异大
- `ts` 等容器也可能能提取，但轨道信息完整度和可预测性通常更差
- 有些字幕轨不能直接无损导出成常见文本格式
- 图片字幕如 PGS、VobSub 需要保留原格式，不能假装导出成 `srt`；如果后续确实需要文本化，只能额外走 OCR 流程
- 语言、标题、forced/default 标记不一定完整

这意味着字幕命名规则必须允许缺省值与去重策略。

### 6.2 BIF 兼容性要先验证

“能生成 BIF”不等于“Emby 一定认”。

需要确认：

- Emby 实际使用的 BIF 格式版本
- 文件命名要求是否固定
- 是否需要配套索引结构
- 不同客户端是否都使用它

建议先做一个最小验证样本。

### 6.3 NFO 范围容易失控

如果只是基于现有文件名、目录结构和本地探测信息输出基础 NFO，会简单很多。

如果要做到像刮削器那样：

- 自动识别电影/剧集
- 访问 TMDB / TVDB / Bangumi 等站点
- 处理多季多集
- 处理女优/系列/番号等特殊规则

这会立刻变成另一个大项目。

但结合你当前链路，文件名与目录已经相对规范，接入 `TMDB` 做受控刮削是可行的。建议第一阶段先做“以 TMDB 为主、规则收敛”的刮削，而不是一开始支持很多站点。

### 6.4 文件监控存在抖动与半写入问题

下载器或复制过程中文件可能尚未完成：

- 监控到创建事件时文件还没写完
- 媒体信息读取可能失败
- 生成的伴生文件可能被重复触发

需要设计：

- 最小静默时间
- 文件大小稳定检测
- 去重和防抖策略

### 6.5 外部工具依赖管理

这类工具本质上会依赖系统中的 CLI：

- `ffmpeg`
- `ffprobe`
- `mkvextract`
- `mediainfo`

需要在 Web 端做：

- 路径配置
- 可执行性检查
- 版本检测
- 错误提示

## 7. 是否有必要替代“神医助手”

结论：有必要替代神医助手，而且应当以“收口你现在分散在多个工具里的生成任务”为主线，先把链路变顺，再逐步接管它当前负责的生成能力。

### 适合先自己做的

- 字幕提取
- `mediainfo.json` 导出
- 基于 `TMDB` 的基础 NFO/actor/剧集组增强
- BIF 生成
- 目录监控与重试机制
- 统一任务编排、日志、失败重试

### 替代原则

- 保留现有 `AniRss + qBittorrent` 下载和命名链路
- 不要求第一版就做成 Emby 插件
- 第一版先覆盖你当前最依赖神医助手的产物生成能力
- 第一版默认不重复生成 Emby 已能稳定落盘的图片
- 每接管一项能力，都要允许和神医助手并行验证一段时间
- 等结果稳定后，再彻底停用神医助手对应功能

### 不适合第一版就追平的

- 复杂刮削
- 深度 Emby 插件集成
- 所有媒体类型与所有特殊命名习惯兼容
- 完整复刻现有商业插件行为
- 一次性迁移掉你现有 `AniRss + Emby + qBittorrent + 神医助手` 的整套工作流

## 8. 路线建议

### 第一阶段

- 用 Go 做独立服务
- Web 端使用 `React + TypeScript`
- 先支持目录监控、字幕提取、mediainfo 导出、NFO 生成、BIF、TMDB 基础增强
- 任务结果写日志并可在页面查看
- 尽量复用你现有目录规范与命名结果，不接管下载器职责
- 以“先替掉神医助手的核心生成能力”为验收目标

### 第二阶段

- 做文件稳定检测、失败重试、去重
- 增强图片接管能力，如 `fanart.tv`、原语言海报、clearlogo、clearart
- 评估 OCR 字幕转换

### 第三阶段

- 评估是否有必要做 Emby 插件
- 如果要做，也尽量只做“触发器/桥接层”，核心仍保留在独立服务

## 9. 当前建议结论

- 先写成独立服务，不先做 Emby 插件
- 开发语言选 Go
- 必须带 Web 端配置
- 前端采用 React 技术栈
- 第一版允许接入 `TMDB` 做受控增强，但不追求多站点和大而全
- 图片默认继续由 Emby 落盘，本项目保留后续接管位
- 第一版目标就是替代神医助手在本地媒体生成链路中的主要职责
- BIF 和 NFO 先做最小可行验证，再确定最终格式

## 12. 剧集一致性检查工具

配套命令入口：`cmd/nyammdcheck`。

示例：

```bash
go run ./cmd/nyammdcheck -config config.yaml -root "D:\\Media\\TV\\Example Show"
```

规则：

- 扫描指定剧集根目录下的视频文件，Season 0 会跳过缺漏判断
- 优先读取同名单集 `.nfo` 获取季度、集数、标题、简介、图片和 provider id，缺失时回退到文件名解析
- 默认读取剧集根目录 `tvshow.nfo` 的 TMDB ID 查询 TMDB，并按 TMDB 该季真实集号判断缺漏；手动 `-tmdb-id` 优先级更高
- TMDB 不可用时，才回退读取季度目录 `season.nfo` 的 `episodecount` 做总量级判断
- 配置 Emby 参数后，会通过 Emby API 对比剧集、季度、单集的标题、简介、图片存在性、可用的 TMDB ID，以及单集文件名

Emby 对比示例：

```bash
go run ./cmd/nyammdcheck -config config.yaml -root "D:\\Media\\TV\\Example Show" -emby-item-url "https://emby.example.com/web/index.html#!/item?id=662" -emby-api-key "你的APIKey"
```

可选参数：

- `-tmdb-id`：手动指定 TMDB 剧集 ID，优先级高于 `tvshow.nfo`
- `-emby-item-url`：Emby 剧集详情页地址，会自动解析服务地址和剧集 ItemId
- `-json`：输出 JSON 报告
- `-fail-on-issue`：发现缺漏、警告或 Emby 差异时以退出码 `2` 结束，便于脚本集成

## 13. 文件对齐检查工具

主程序页面入口：`剧集核对` 页面里的 `文件对齐检查` 卡片。

该能力独立于剧集元数据检查，用于核对本地目录和远端目录的文件是否对齐。远端访问通过接口抽象，当前实现为 SFTP，后续可以补充其他远端文件系统。

规则：

- 默认检查多文件、少文件、直接匹配文件的大小差异
- 默认允许本地视频文件匹配远端同名 `.strm` 文件，此时不会比较视频与 `.strm` 的文件大小
- 开启 `比较 MD5` 后，会对直接匹配的文件计算并比较 MD5；视频匹配 `.strm` 时不会做 MD5
- SFTP 默认要求提供 `known_hosts`，如需临时跳过主机指纹校验可显式开启 `跳过 SFTP 主机指纹校验`
- API 入口为 `POST /api/audit/files`，供主页面调用
